# LLM-Free GraphRAG 증류 프로젝트 — 세부 Sub-Phase 분해 (개정판)

`graphrag_subprojects_phases.md`(개정판)의 각 Phase를 실행 단위(1개 세션 or 1일 내 완료 가능한 작업)로 한 단계 더 쪼갠 문서. 표기법: `Phase.하위번호` (예: `1.1-a`). 각 항목에 예상 소요(S=반나절 이하, M=1일, L=2~3일)와 완료 기준(Done when)을 명시.

> **리소스 전제**: 단일 RTX 3090(24GB, 시간 무제한) + API 예산 사실상 $0. 교사 모델은 로컬 vLLM 서빙 강모델(예: Qwen2.5-32B-Instruct-AWQ). baseline 인덱싱도 동일 로컬 모델로 재현. 상세: `graphrag_llmfree_design.md`.

---

## Phase 0 — 프로젝트 공통 착수 (1주차, 서브1과 병렬)

### Phase 0.0 — 로컬 교사 모델 서빙 환경 구축
- **0.0-a** (M) vLLM 설치, CUDA/드라이버 호환성 확인 (RTX 3090, 24GB VRAM 기준)
- **0.0-b** (M) Qwen2.5-32B-Instruct-AWQ(또는 14B-Instruct, 속도 우선 시) 다운로드 및 4bit 양자화 로드 확인
- **0.0-c** (S) OpenAI 호환 엔드포인트로 vLLM 서버 기동, curl/Python 클라이언트로 샘플 요청 1건 응답 확인
- **0.0-d** (M) 처리량 벤치마크: 20개 청크로 초당 토큰 수, 청크당 평균 소요시간 측정 → 이후 Phase 1.2/3.1 규모 산정의 기초 자료로 기록
- Done when: 로컬 엔드포인트가 안정적으로 응답하고, 처리량 수치가 문서화됨

### Phase 0.5 — Baseline 5종 재현 착수 (1주차부터 서브1과 완전 병렬)
- **0.5-a** (L) Microsoft GraphRAG 설치, 인덱싱 LLM 호출 지점을 Phase 0.0의 로컬 vLLM 엔드포인트로 교체 (config의 API base URL 변경)
- **0.5-b** (M) LightRAG 설치, 동일하게 로컬 엔드포인트 연동
- **0.5-c** (M) LiteSemRAG 재현 (LLM-free 계열, 임베딩 모델만 필요)
- **0.5-d** (M) dependency-parsing(arXiv:2507.03226) 재현 — 코드 공개 여부 먼저 확인, 없으면 논문 기술만으로 최소 재현
- **0.5-e** (M) NoLLMRAG 재현
- **0.5-f** (S) 공통 QA 인터페이스(질의→검색→응답 함수 시그니처) 정의, 5개 wrapper 모두 이 인터페이스로 감싸기
- Done when: 5개 baseline 모두 샘플 질의 1건에 정상 응답, 재현 중 발견된 리스크(의존성 충돌 등) 로그로 기록

## 서브프로젝트 1 — 합성 데이터 생성 & 검증 파이프라인

### Phase 1.0 — 환경 & 스키마 준비
- **1.0-a** (S) UltraDomain HF `datasets` 로더 스크립트 작성, 18개 도메인 서브셋 목록 확인
- **1.0-b** (S) DocRED 로더 스크립트 작성 (train_annotated/dev/test split 구조 확인, relation id ↔ 이름 매핑 테이블 로드)
- **1.0-c** (S) Triple 스키마 `(entity1, entity1_type, relation, entity2, entity2_type, source_span, confidence)` 를 Pydantic/JSON Schema로 고정, 버전 태그(`schema_v1`) 부여
- **1.0-d** (M) 로컬 교사 모델(Phase 0.0 엔드포인트) 호출 wrapper 작성: 배치 큐잉, 실패 시 재시도, 호출별 소요시간 로깅(비용 대신 GPU-시간 추적)
- Done when: 두 데이터셋 모두 100개 샘플을 로컬에 캐시하고, 스키마 검증 유닛테스트 통과

### Phase 1.1 — 프롬프트 설계 & 파일럿
- **1.1-a** (M) few-shot 예시 3~5개 수작업 제작 (도메인 다양성 고려: 서사/기술/금융 각 1개 이상)
- **1.1-b** (S) 출력 강제 포맷(JSON mode 또는 구조화 출력) 적용, 파싱 실패시 재시도 로직(최대 2회) 구현
- **1.1-c** (M) 20~30개 청크로 파일럿 실행, 실패율/파싱 오류 유형 수기 분류
- **1.1-d** (S) 파일럿 결과 기반 프롬프트 1차 개정 (오류 패턴 반영)
- Done when: 파일럿 파싱 성공률 ≥95%, 실패 사례 로그 문서화

### Phase 1.2 — UltraDomain 대량 생성
- **1.2-a** (S) 청크 샘플링 전략 확정 (도메인별 균등 vs 랜덤) — Phase 0.0 처리량 벤치마크 기준으로 500~1,000개가 GPU 시간상 현실적인지 재확인
- **1.2-b** (M) 배치 실행 파이프라인: 체크포인트 저장(N개마다), 중단 후 재개 가능하도록 설계
- **1.2-c** (S) 청크 해시 기반 캐싱 (동일 청크 재요청 방지)
- **1.2-d** (L) 500~1,000개 청크 실제 실행 (GPU-시간 추적, API 비용 없음)
- **1.2-e** (S) 원본 청크 ↔ 생성 triple 매핑을 parquet/JSON lines로 저장
- **1.2-f** (S) 도메인별 균등 샘플 50~100개 청크를 **고정 평가 서브셋**으로 별도 태깅·저장 (Phase 3.7에서 재사용)
- Done when: 목표 청크 수 완료 + 고정 평가 서브셋 확정, 실행 로그(총 GPU-시간, 실패율) 요약 리포트 존재

### Phase 1.3 — DocRED 대조군 실행 & 관계어 정렬
- **1.3-a** (S) DocRED 문서를 UltraDomain 청크와 동일 길이 단위로 재분할 (공정 비교를 위해)
- **1.3-b** (M) 동일 프롬프트/파이프라인으로 DocRED 문서 실행 (표본 수는 1.2와 비례하거나 전체 test split)
- **1.3-c** (S) 엔티티 alias resolution: DocRED의 coreference 정보와 교사 모델 출력 엔티티 문자열 정렬 로직
- **1.3-d** (M) **(신설) 관계어 정렬**: 교사 모델의 자유생성 관계어(예: "founded by")를 DocRED 96종 스키마(예: P112)로 매핑하는 로직 구현. 방식: (a) 수작업 매핑 테이블 우선 작성 후 (b) 커버 안 되는 관계어는 임베딩 유사도로 최근접 DocRED 관계에 매핑, 매핑 신뢰도 낮으면 "unmapped"로 분리
- **1.3-e** (S) 매핑 커버리지 확인 (전체 교사 관계어 중 몇 %가 DocRED 스키마로 매핑되었는지)
- Done when: DocRED 문서 예측 결과 JSON 저장 + 관계어 정렬 매핑 테이블 + 매핑 커버리지 리포트 완성

### Phase 1.4 — 정확도 상한선 정량화
- **1.4-a** (S) 매칭 규칙 정의: exact match vs relation-type-only match vs partial(엔티티만 일치) 3단계로 분리 채점 (정렬된 관계어 기준)
- **1.4-b** (M) Precision/Recall/F1 계산 스크립트, 도메인·relation type별 breakdown
- **1.4-c** (S) 오류 유형 수동 샘플링(50건) 분류: hallucination / schema 불일치 / relation 방향 오류 / 매핑 실패 등
- **1.4-d** (S) 이 수치를 "닫힌 스키마 RE 벤치마크 참고 성능"으로 명확히 라벨링 (UltraDomain 자체 품질은 Phase 3.7에서 별도 검증됨을 리포트에 명시)
- Done when: 정량 리포트 + 오류 유형 분포 표 완성

### Phase 1.5 — 필터링 규칙 설계
- **1.5-a** (S) Rule 기반 필터 구현: 빈 triple, self-loop, 타입 불일치 제거
- **1.5-b** (M) confidence 산출: 동일 청크 2~3회 재생성(로컬 모델이라 API 비용 없이 반복 가능) 후 self-consistency 일치도로 계산
- **1.5-c** (M) 통계 기반 필터: relation 빈도 이상치, confidence 임계값 스윕(0.3/0.5/0.7) 별 영향 측정
- **1.5-d** (S) 필터링 모듈 유닛테스트 (edge case: 빈 문자열, 중복 대소문자 등)
- Done when: 필터 모듈 코드 + confidence 산출 로직 문서 + 임계값별 영향 로그

### Phase 1.6 — 필터 전후 비교 & 최종 데이터셋 확정
- **1.6-a** (S) 필터 적용 전/후 데이터 규모, 노이즈 비율 비교표
- **1.6-b** (S) train/val split (도메인 균형 고려한 stratified split, 고정 평가 서브셋은 train에서 제외)
- **1.6-c** (S) 데이터셋 버전 태깅 및 카드(datasheet) 작성: 생성 모델(로컬 Qwen2.5 버전), 날짜, 규모, 알려진 한계(로컬 교사가 proprietary 대비 약할 수 있음 명시)
- Done when: 최종 JSON 데이터셋 파일 + datasheet 문서 존재, 서브프로젝트 2에 바로 투입 가능

## 서브프로젝트 2 — 경량 추출 모델 SFT (증류)

### Phase 2.0 — 데이터 통합 설계
- **2.0-a** (S) SciERC/WebNLG/FewRel을 공통 triple 스키마(entity_type, source_span 포함)로 변환하는 어댑터 작성
- **2.0-b** (S) 라이선스 확인 (각 데이터셋 재배포/연구용 사용 조건 체크 — 논문에 명시할 것)
- **2.0-c** (S) 혼합 비율 실험 설계표 작성 (0%/10%/30%/50%, 각 조합의 총 샘플 수 계산)
- Done when: 통합 로더 하나로 4개 데이터소스 모두 동일 포맷 산출 확인

### Phase 2.1 — 모델/학습 인프라
- **2.1-a** (M) DeBERTa-v3 span-based 헤드 vs 1B급 seq2seq 두 후보 최소 구현체 프로토타입 (더미 데이터로 forward pass만 확인, 둘 다 3090 24GB에 여유 있게 적재됨)
- **2.1-b** (S) 선택 기준 명문화: 추론 속도, 정확도, 파인튜닝 난이도 3축 비교 기준표
- **2.1-c** (M) LoRA/QLoRA 설정(rank, alpha, target modules) 및 학습 스크립트 골격, config 기반 재현 가능하게
- **2.1-d** (S) wandb(무료 tier) 로깅 연동, 재현용 seed 고정
- Done when: 학습 스크립트가 100샘플 더미로 1 epoch 정상 종료

### Phase 2.2 — 1차 학습 (주 데이터만)
- **2.2-a** (M) 서브프로젝트 1 데이터만으로 전체 학습 실행 (단일 GPU 순차 실행 시간 기록)
- **2.2-b** (S) Loss curve 검토, overfitting 징후 체크 (train/val gap)
- **2.2-c** (S) 조기 정성 검증: 임의 5개 문장에 대해 직접 추론 결과 육안 확인
- Done when: 베이스라인 체크포인트 저장 + 학습 로그 존재

### Phase 2.3 — 보조 데이터 비율 실험
- **2.3-a** (M) 각 비율(0/10/30/50%)별 반복 학습 실행 (동일 시드, 동일 epoch 수로 통제, 단일 GPU라 순차 실행 — 총 소요시간 사전 추정해 일정에 반영)
- **2.3-b** (S) 비율별 검증 성능 비교표 + 최적 비율 선정 근거
- Done when: 비율별 체크포인트 4개 + 비교 로그

### Phase 2.4 — DocRED 최종 평가
- **2.4-a** (S) 최적 비율 모델을 DocRED test split에 F1 평가 (Phase 1.3-d와 동일한 관계어 정렬 로직 재사용)
- **2.4-b** (S) Phase 1.4 교사 모델 상한선과 나란히 비교하는 표 생성 → "교사 대비 학생 성능 유지율(%)" 확정 (핵심 결과물)
- Done when: 유지율 수치 확정

### Phase 2.5 — FewRel 일반화 테스트
- **2.5-a** (M) 미학습 relation 타입에 대한 zero-shot/few-shot 평가 실행
- **2.5-b** (S) 실패 사례(관계 유형별) 분석, 어떤 관계 카테고리가 특히 약한지 정리
- Done when: 일반화 리포트 + 약점 relation 유형 목록

### Phase 2.6 — 최종 모델 확정 & 문서화
- **2.6-a** (S) 최종 체크포인트 선정 근거 정리 (비율, 성능, 속도 종합)
- **2.6-b** (S) 모델 카드 작성 (학습 데이터, 하이퍼파라미터, 한계, 라이선스, 교사 모델이 로컬 오픈소스임을 명시)
- Done when: 공개 가능한 체크포인트 + 모델 카드 + 최종 벤치마크 표 확정 → 서브프로젝트 3 착수 가능

## 서브프로젝트 3 — 그래프 구축 & 커뮤니티 요약

### Phase 3.0 — 추론 파이프라인 구축
- **3.0-a** (M) 배치 추론 스크립트 (긴 문서 청킹 전략 포함), GPU 배치 크기 튜닝
- **3.0-b** (S) 처리량 벤치마크 (문서/초, GPU 메모리 사용량) 기록 — GPU-시간 비용 비교 근거자료로 재사용
- Done when: 10개 문서로 end-to-end 추론 smoke test 통과

### Phase 3.1 — 전체 코퍼스 Triple 추출
- **3.1-a** (L) UltraDomain 전체(428개 교재) 배치 실행, 중간 체크포인트/재개 지원
- **3.1-b** (S) 원시 triple에 문서/도메인 메타데이터 태깅, Phase 1.2-f 고정 평가 서브셋 문서는 별도 플래그 표시
- Done when: 전체 코퍼스 triple 원시 데이터셋 완성

### Phase 3.2 — 엔티티 정규화 & 중복 제거
- **3.2-a** (S) 문장 임베딩 모델 선정 및 엔티티 문자열 임베딩 생성
- **3.2-b** (M) 유사도 기반 클러스터링/병합 로직 구현 (entity_type이 다르면 병합 후보에서 제외, 임계값 스윕으로 최적점 탐색)
- **3.2-c** (S) 병합 오류 샘플 검수 (over-merge/under-merge 사례 육안 확인 20건)
- Done when: 정규화된 엔티티 매핑 테이블 + 병합 정밀도 수기 검증 결과

### Phase 3.3 — 그래프 DB 구축
- **3.3-a** (S) Phase 3.35(검색 인터페이스)에 필요한 쿼리 패턴(1-hop/2-hop 탐색, 커뮤니티 조회)을 먼저 목록화한 뒤 NetworkX 스키마(노드/엣지 속성) 확정
- **3.3-b** (M) Triple → 그래프 적재 스크립트, 대규모 시 Neo4j 마이그레이션 스크립트
- Done when: 그래프 인스턴스 로드 + 기본 통계(노드/엣지 수) 출력 확인

### Phase 3.35 — 검색(retrieval) 인터페이스 구축 (신설)
- **3.35-a** (M) 쿼리 → 서브그래프 탐색(1-hop/2-hop) 로직 구현
- **3.35-b** (M) 서브그래프 랭킹 기준 구현 (엔티티 매칭 점수, degree 기반 중요도 등)
- **3.35-c** (S) 컨텍스트 구성(서브그래프 → 텍스트 프롬프트 변환) 로직
- **3.35-d** (S) 최종 응답 생성은 로컬 LLM 재활용(또는 별도 응답 생성기) — 인덱싱만 LLM-free임을 코드 주석/문서에 명시
- Done when: 쿼리 1건에 대해 end-to-end(쿼리→서브그래프→응답) 테스트 통과

### Phase 3.4 — 커뮤니티 탐지
- **3.4-a** (S) Leiden/Louvain 파라미터(resolution) 스윕
- **3.4-b** (S) 커뮤니티 크기 분포 시각화, 지나치게 크거나 작은 커뮤니티 확인
- Done when: 최종 resolution 값 확정 + 라벨링된 그래프 저장

### Phase 3.5 — 커뮤니티 요약 생성
- **3.5-a** (M) 커뮤니티 소속 엔티티/관계가 언급된 원문 문장 풀을 구성한 뒤 TextRank(또는 대안) 추출 요약 구현 (커뮤니티=엔티티/관계 집합이므로 문장 풀로 변환하는 단계가 선행되어야 함)
- **3.5-b** (S) 커뮤니티별 요약 품질 수기 샘플 검토(10~15개) — LLM 요약과의 정성적 격차를 미리 파악해 논문 Limitation에 반영할 근거 확보
- Done when: 커뮤니티별 요약 텍스트 저장 + 정성 검토 메모

### Phase 3.6 — 구조 통계 비교
- **3.6-a** (S) 노드 수/엣지 수/평균 degree/커뮤니티 수 산출
- **3.6-b** (S) 기존 baseline 논문에 보고된 UltraDomain 그래프 통계와 나란히 비교표 작성
- Done when: 비교표 + 구조적 유사성에 대한 짧은 해석 문단 (※ 이는 recall이 아니라 구조 비교임을 리포트에 명확히 표기)

### Phase 3.7 — 그래프 커버리지 Recall 계산 (신설)
- **3.7-a** (S) Phase 1.2-f 고정 평가 서브셋(교사 모델 참조 triple)과 Phase 3.1의 학생 모델 triple 중 동일 서브셋을 매칭
- **3.7-b** (M) Phase 1.4와 동일한 매칭 규칙으로 엔티티 recall, 관계 recall 계산
- **3.7-c** (S) 도메인별 recall 분포 확인 (특정 도메인에서 유독 낮은지)
- Done when: 그래프 커버리지 recall 리포트 완성 — 설계서 §6.3 "품질" 축 핵심 수치 확정

### Phase 4.4 — 비용(GPU-시간) 측정

## 서브프로젝트 5 — 도메인 적응 실험

### Phase 5.0 — Fin/Leg 서브셋 준비
- **5.0-a** (S) UltraDomain Fin/Leg 서브셋 분리, train/eval split
- **5.0-b** (S) 스키마 정합성 확인 (서브프로젝트 1~2와 동일 포맷인지)
- Done when: Fin/Leg 학습·평가 데이터셋 파일 확정

### Phase 5.1 — Zero-shot 베이스라인
- **5.1-a** (M) 범용 모델(서브2 2.6 산출물)을 Fin/Leg에 그대로 추론
- **5.1-b** (S) 라벨 없는 경우 대비 정성 평가 기준(수기 샘플 채점) 마련
- Done when: zero-shot 성능 결과 확정

### Phase 5.2 — 도메인 특화 파인튜닝
- **5.2-a** (M) Fin, Leg 각각 별도 어댑터로 추가 SFT
- **5.2-b** (S) 소규모 하이퍼파라미터 탐색 (learning rate 2~3개 값)
- Done when: Fin/Leg 전용 체크포인트 각 1개 확정

### Phase 5.3 — 재평가 & 격차 정량화 (triple 레벨)
- **5.3-a** (S) 동일 평가셋에서 zero-shot vs adapted 비교
- **5.3-b** (S) 통계적 유의성 검정 (bootstrap 또는 t-test), 표본 수가 작을 경우 신뢰구간으로 대체
- Done when: triple 레벨 ablation 표 + 유의성 검정 결과 확정

### Phase 5.4 — QA 레벨 도메인 적응 검증 (신설)
- **5.4-a** (M) Fin/Leg 도메인 특화 질문셋 구성 (GraphRAG-Bench 내 해당 도메인 질문 재사용 우선, 부족하면 소규모 수작업 제작)
- **5.4-b** (M) 서브4(Phase 4.1~4.2)의 QA 평가 하네스를 재사용해 zero-shot vs adapted 모델의 QA 정확도 비교
- **5.4-c** (S) triple 레벨 격차(5.3)와 QA 레벨 격차(5.4)를 나란히 비교해 "triple 성능 향상이 실제 QA 향상으로 이어지는가" 해석
- Done when: 최종 ablation 표(triple + QA 레벨 모두 포함) 확정 — 논문 기여 C2의 핵심 근거

|---|---|
