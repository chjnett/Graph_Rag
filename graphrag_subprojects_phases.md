# LLM-Free GraphRAG 증류 프로젝트 — 세부 Phase 분해 (개정판)

`graphrag_subprojects_datasets.md`의 5개 서브프로젝트를 실행 가능한 소단위 phase로 분해한 문서. 각 phase는 "입력 → 작업 → 산출물"이 명확한 최소 실행 단위로 구성.

> **리소스 전제**: 단일 RTX 3090(24GB, 시간 무제한) + API 예산 사실상 $0. "교사 모델"은 로컬 vLLM 서빙 강모델(예: Qwen2.5-32B-Instruct-AWQ)을 의미. 상세는 `graphrag_llmfree_design.md` 참고.
>
> **개정 사항 요약** (검토에서 발견된 이슈 반영):
> - Triple 스키마를 `(entity1, entity1_type, relation, entity2, entity2_type, source_span, confidence)`로 전 phase 통일
> - Phase 1.3~1.4에 DocRED 관계어 정렬(mapping) 단계 추가 — 정렬 없이는 교사 정확도 상한선 비교가 무효함
> - **Baseline 재현(구 Phase 4.0)을 Phase 0.5로 승격해 1주차부터 서브1과 병렬 착수** — 가장 리스크가 큰 항목을 최후반에 두지 않음
> - 그래프 커버리지 recall을 실제로 산출하는 Phase 3.7 신설 (구조 통계 비교만으로는 §6.3 지표를 채울 수 없었음)
> - 검색(retrieval) 인터페이스 구축 Phase 3.35 신설 (기존엔 어떤 phase에도 없었음)
> - 서브5에 Phase 5.4(QA 레벨 도메인 적응 검증) 추가

---

## Phase 0 — 프로젝트 공통 착수 (신설, 병렬 트랙)

### Phase 0.0 — 로컬 교사 모델 서빙 환경 구축
- RTX 3090에 vLLM 설치, Qwen2.5-32B-Instruct(또는 14B, 속도 우선 시) 4bit 양자화(AWQ/GPTQ) 모델 로드
- OpenAI 호환 엔드포인트로 기동, 샘플 요청 1건 응답 확인
- 처리량 벤치마크 (초당 토큰, 청크당 평균 소요시간) — 이후 전체 일정 산정의 기초 자료
- **산출물**: 로컬 서빙 기동 스크립트, 처리량 벤치마크 로그

### Phase 0.5 — Baseline 5종 재현 착수 (구 Phase 4.0, 1주차부터 서브1과 병렬)
- Microsoft GraphRAG, LightRAG의 인덱싱 LLM 호출을 Phase 0.0의 로컬 vLLM 엔드포인트로 연결 (OpenAI API 키 자리에 로컬 엔드포인트 URL 지정)
- LiteSemRAG, dependency-parsing(arXiv:2507.03226), NoLLMRAG 재현/wrapper 작성 (LLM-free 계열이라 로컬 모델 불필요)
- 공통 QA 인터페이스 정의 (질의→검색→응답 함수 시그니처 통일) — 서브4에서 재사용
- **왜 최우선인가**: 외부 코드 재현은 의존성 충돌, 미문서화된 하이퍼파라미터, 청킹 방식 차이 등으로 가장 예측 불가능한 작업. 서브3(그래프 구축) 완료를 기다리면 문제 발견이 프로젝트 후반부로 밀려 전체 일정이 위태로워짐
- **산출물**: 5개 baseline wrapper (로컬 LLM 연동 포함) + 공통 QA 인터페이스 코드, 샘플 질의 1건 정상 응답 확인

## 서브프로젝트 1 — 합성 데이터 생성 & 검증 파이프라인

### Phase 1.0 — 환경 & 스키마 준비
- UltraDomain, DocRED 다운로드 스크립트 작성 (HuggingFace `datasets` 라이브러리)
- Triple 스키마 확정: `(entity1, entity1_type, relation, entity2, entity2_type, source_span, confidence)` — entity_type(엔티티 정규화/그래프 노드 타입에 필요)과 source_span(환각률 검증에 필요) 둘 다 포함
- 로컬 교사 모델 호출 wrapper 작성 (Phase 0.0의 vLLM 엔드포인트 대상, rate limit 대신 GPU 큐잉/배치 크기로 처리량 관리)
- **산출물**: 데이터 로더 스크립트, 스키마 문서(JSON Schema)

### Phase 1.1 — 프롬프트 설계 & 파일럿
- 청크→triple 추출 프롬프트 초안 작성 (few-shot 예시 포함)
- 소규모 파일럿(20~30개 청크)으로 프롬프트 반복 개선
- 출력 파싱 로직(JSON/구조화 출력) 구현 및 실패율 측정
- **산출물**: 확정 프롬프트, 파서 코드, 파일럿 결과 노트

### Phase 1.2 — UltraDomain 대량 생성
- 청크 샘플링 전략 결정 (도메인별 균등 샘플링 vs 랜덤, 500~1,000개 — Phase 0.0 처리량 벤치마크로 실행 가능 규모 재확인)
- 배치 실행 파이프라인 (재시도/캐싱/체크포인트 포함)
- 원본 청크 ↔ 생성 triple 매핑 저장
- 이 중 도메인별 균등 샘플 50~100개 청크를 **고정 평가 서브셋**으로 별도 표시 (Phase 3.7 커버리지 recall 계산에 재사용)
- **산출물**: 합성 triple 데이터셋(JSON, 원시 버전) + 고정 평가 서브셋 목록

### Phase 1.3 — DocRED 대조군 실행 & 관계어 정렬
- 동일 프롬프트/파이프라인으로 DocRED 문서에 교사 모델 적용
- DocRED 정답 라벨과 엔티티 alias resolution 정렬
- **(신설) 관계어 정렬(mapping)**: 교사 모델이 자유생성한 관계어(open-vocabulary)를 DocRED 96종 닫힌 스키마로 매핑하는 로직 구현 (예: 임베딩 유사도 기반 완화 매칭, 또는 수작업 매핑 테이블). 이 단계 없이 다음 Phase 1.4로 넘어가면 안 됨 — 스키마 불일치로 정확도가 인위적으로 낮게 나옴
- **산출물**: DocRED 위 교사 모델 예측 결과(JSON) + 관계어 정렬 매핑 테이블

### Phase 1.4 — 정확도 상한선 정량화
- 정렬된 교사 모델 예측 vs DocRED gold label 매칭 로직 (exact match / relation-type match)
- Precision / Recall / F1 계산
- 이 수치는 "닫힌 스키마 RE 벤치마크에서의 참고 성능"으로 한정 — UltraDomain 자체에서의 품질은 Phase 3.7의 recall로 별도 검증됨을 명시
- **산출물**: 교사 모델 상한선 리포트 (수치 + 오류 유형 분석)

### Phase 1.5 — 필터링 규칙 설계
- 노이즈 패턴 분석 (중복 triple, 저신뢰 관계, 순환 관계 등)
- confidence 산출 방식 확정: 동일 청크 2~3회 재생성 후 self-consistency(일치도)로 계산 (자기보고 점수 대신 — 로컬 모델이라 반복 호출이 API 비용 없이 가능)
- 규칙 기반 필터 + confidence 임계값 필터 구현
- **산출물**: 필터링 모듈 코드, confidence 산출 로직 문서

### Phase 1.6 — 필터 전/후 비교 & 최종 데이터셋 확정
- 필터 적용 전/후 노이즈 비율, 데이터 규모 변화 비교
- 최종 학습용 데이터셋 버전 고정(train/val split)
- **산출물**: 최종 합성 데이터셋(JSON) + 품질 리포트(정밀도/재현율 vs DocRED, 필터 효과)

## 서브프로젝트 2 — 경량 추출 모델 SFT (증류)

### Phase 2.0 — 데이터 통합 설계
- 서브1 산출물(주 데이터) + SciERC/WebNLG/FewRel(보조 데이터) 포맷 통일 (동일 스키마로)
- 보조 데이터 혼합 비율 실험안 설계 (예: 0%, 10%, 30%, 50%)
- **산출물**: 통합 데이터 로더, 비율 실험 계획표

### Phase 2.1 — 모델/학습 인프라 셋업
- 베이스 모델 선정 확정 (DeBERTa-v3 vs 1B급 seq2seq 비교 기준 정의 — 둘 다 단일 3090에 여유 있게 적재됨)
- QLoRA/LoRA 설정 (rank, target modules), 학습 스크립트 골격 작성
- **산출물**: 학습 파이프라인 코드(재현 가능한 config 포함)

### Phase 2.2 — 베이스라인 학습 (주 데이터만)
- 서브1 데이터만으로 1차 SFT 실행
- Loss curve, 초기 성능 확인 (샘플 검증)
- **산출물**: 베이스라인 체크포인트

### Phase 2.3 — 보조 데이터 비율 실험
- Phase 2.0에서 설계한 비율별로 반복 학습 (단일 GPU 순차 실행, 소요시간 사전 벤치마크로 일정 반영)
- 각 비율에서 검증셋 성능 비교
- **산출물**: 비율별 체크포인트 + 비교 로그

### Phase 2.4 — DocRED test 평가
- 최적 비율 모델을 DocRED test split에서 F1 평가 (학생 모델 예측도 Phase 1.3과 동일한 관계어 정렬 로직 적용)
- 교사 모델 상한선(Phase 1.4)과 비교 → "교사 대비 학생 성능 유지율(%)" 산출
- **산출물**: DocRED 벤치마크 성능표

### Phase 2.5 — FewRel 일반화 테스트
- 미학습 관계 유형에 대한 zero-shot/few-shot 성능 측정
- 실패 사례(관계 유형별) 분석
- **산출물**: FewRel 일반화 리포트

### Phase 2.6 — 최종 모델 확정 & 문서화
- 최종 체크포인트 선정 근거 정리
- 모델 카드(학습 데이터, 하이퍼파라미터, 한계) 작성
- **산출물**: 학생 모델 체크포인트(공개용) + DocRED/FewRel 벤치마크 성능표(최종)

## 서브프로젝트 3 — 그래프 구축 & 커뮤니티 요약

### Phase 3.0 — 추론 파이프라인 구축
- 서브2 체크포인트 로드, 배치 추론 스크립트 (긴 문서 청킹 전략 포함)
- 처리량/GPU-시간 산정 (§6.3 비용 지표의 기초 자료)
- **산출물**: 추론 파이프라인 코드

### Phase 3.1 — 전체 코퍼스 Triple 추출
- UltraDomain 전체(428개 교재)에 대해 학생 모델 추론 실행
- 원시 triple 저장 (문서/도메인 메타데이터 포함), Phase 1.2의 고정 평가 서브셋에 해당하는 문서는 결과를 별도로 표시
- **산출물**: 원시 triple 대규모 데이터셋

### Phase 3.2 — 엔티티 정규화 & 중복 제거
- 엔티티 임베딩 생성 (문장 임베딩 모델, entity1_type/entity2_type 필드 활용해 타입 다른 엔티티는 병합 후보에서 제외)
- 유사도 기반 클러스터링/병합 로직 구현 및 임계값 튜닝
- **산출물**: 정규화된 엔티티 매핑 테이블

### Phase 3.3 — 그래프 DB 구축
- NetworkX(프로토타입) 또는 Neo4j(운영) 스키마 설계 — **Phase 3.35(검색 인터페이스)에서 필요한 쿼리 패턴(1-hop/2-hop 탐색, 커뮤니티 단위 조회)을 먼저 확인한 뒤 스키마에 반영**
- Triple → 그래프 적재 파이프라인
- **산출물**: 그래프 DB 인스턴스 + 로딩 스크립트

### Phase 3.35 — 검색(retrieval) 인터페이스 구축 (신설)
- 쿼리 → 관련 서브그래프 탐색(1-hop/2-hop) → 컨텍스트 구성 로직 구현 (`retrieval/graph_retriever.py`)
- 서브그래프 랭킹 기준 정의 (예: 엔티티 매칭 점수, degree 기반 중요도)
- 최종 응답 생성은 LLM 사용(로컬 모델 재활용 가능) — 인덱싱만 LLM-free이고 응답 생성 단계는 표준 GraphRAG 구조와 동일함을 코드 주석/문서에 명시
- **산출물**: 검색 인터페이스 코드, 쿼리 1건 end-to-end 테스트 통과
- **왜 필요한가**: 기존 phase 계획에는 이 컴포넌트가 없었으나, 서브4의 QA 평가가 성립하려면 반드시 필요함

### Phase 3.4 — 커뮤니티 탐지
- Leiden/Louvain 알고리즘 적용, 해상도 파라미터 튜닝
- 커뮤니티 크기 분포 확인
- **산출물**: 커뮤니티 라벨링된 그래프

### Phase 3.5 — 커뮤니티 요약 생성
- 추출 요약 기법(TextRank 등)을 **커뮤니티 소속 엔티티/관계가 언급된 원문 문장 풀**에 적용 (커뮤니티는 문장 집합이 아니라 엔티티/관계 집합이므로, 적용 대상을 명확히 "구성 문서의 관련 문장들"로 정의)
- 요약 품질 샘플 점검 (수동 검토, 10~15개) — LLM 요약 대비 정성적 격차를 미리 파악해 논문 Limitation에 반영
- **산출물**: 커뮤니티별 요약 텍스트 + 정성 검토 메모

### Phase 3.6 — 구조 통계 비교
- 노드 수/엣지 수/평균 degree/커뮤니티 수 등 산출
- 기존 baseline 논문(GraphRAG/LightRAG 등)의 UltraDomain 그래프 통계와 비교
- **산출물**: 구조 통계 비교표 (※ 이는 "구조적 유사성"이며 아래 Phase 3.7의 "추출 정확도"와는 다른 지표임에 유의)

### Phase 3.7 — 그래프 커버리지 Recall 계산 (신설)
- Phase 1.2의 고정 평가 서브셋(50~100청크)에 대해, 교사 모델이 만든 참조 triple(서브1 산출물)과 학생 모델이 만든 triple(Phase 3.1)을 동일 문서 단위로 비교
- 엔티티 recall, 관계 recall 계산 (Phase 1.4와 동일한 매칭 규칙 재사용)
- **산출물**: 그래프 커버리지 recall 리포트 — 설계서 §6.3 "품질" 축의 핵심 수치
- **왜 필요한가**: Phase 3.6의 구조 통계 비교만으로는 "같은 관계를 뽑았는가"를 answer할 수 없음. 이 phase가 있어야 논문이 약속한 지표가 실제로 채워짐

## 서브프로젝트 4 — 다운스트림 QA 평가

> Baseline 재현(구 Phase 4.0)은 **Phase 0.5로 이동**해 1주차에 이미 착수됨. 아래는 서브3 완료 후 실행되는 QA 평가 실행 단계.

### Phase 4.1 — GraphRAG-Bench 난이도별 평가
- 4단계 난이도(사실검색/복합추론/맥락요약/창의생성) 별 질문 분리
- 우리 방법(Phase 3.35 검색 인터페이스 사용) vs baseline(Phase 0.5 wrapper) 각각 실행 및 채점 기준 확정
- 다중 시드(최소 3회) 또는 부트스트랩으로 신뢰구간 산출 — 단일 실행 비교는 지양
- **산출물**: 난이도별 성능 원시 결과 + 신뢰구간

### Phase 4.2 — HotpotQA 인용 정확도 평가
- Supporting facts 대비 근거 문단(=검색된 서브그래프의 출처 source_span) 일치율 계산 로직 구현
- 실행 및 결과 수집
- **산출물**: 인용/근거 일치율 리포트

### Phase 4.3 — MultiHop-RAG 도메인 일반화 평가
- 뉴스 도메인(UltraDomain 밖)에서 동일 파이프라인 실행
- UltraDomain 내부 성능과 도메인 밖 성능 비교
- **산출물**: 도메인 일반화 성능 리포트

### Phase 4.4 — 비용(GPU-시간) 측정
- 우리 파이프라인(서브1~3 전체 GPU-시간) vs baseline(Phase 0.5, 로컬 LLM 인덱싱 GPU-시간)을 동일 단위로 비교
- 참고용으로 동일 작업을 유료 API로 했을 경우의 추정 $ 비용도 병기(선택)
- **산출물**: 비용 비교 데이터

### Phase 4.5 — 종합 테이블 & 그래프 작성
- 논문용 Table(정확도/비용 비교, 신뢰구간 포함) 및 난이도별 성능 분해 그래프 작성
- **산출물**: 최종 비교 Table + 그래프(논문 삽입용)

## 서브프로젝트 5 — 도메인 적응 실험

### Phase 5.0 — Fin/Leg 서브셋 준비
- UltraDomain 내 Fin/Leg 서브셋 분리 및 전처리
- 적응용 학습 데이터 / 평가 데이터 split 설계, 스키마 정합성 확인
- **산출물**: Fin/Leg 학습·평가 데이터셋

### Phase 5.1 — Zero-shot 베이스라인 평가
- 서브2 범용 모델을 Fin/Leg에 그대로 적용
- Triple 추출 성능(가능하면 라벨 기반, 없으면 샘플 수동 평가) 측정
- **산출물**: Zero-shot 성능 결과

### Phase 5.2 — 도메인 적응 파인튜닝
- Fin/Leg 일부 데이터로 추가 SFT(LoRA 계속 학습 or 신규 어댑터)
- 하이퍼파라미터 소규모 탐색
- **산출물**: 도메인 적응 체크포인트(Fin, Leg 각각)

### Phase 5.3 — 재평가 & 격차 정량화 (triple 레벨)
- 적응 모델을 동일 평가셋에서 재평가
- Zero-shot 대비 성능 격차 계산 및 통계적 유의성 확인 (bootstrap 또는 t-test)
- **산출물**: 도메인 적응 전/후 triple 성능 비교표

### Phase 5.4 — QA 레벨 도메인 적응 검증 (신설)
- Fin/Leg 도메인 특화 질문셋 구성 (GraphRAG-Bench에 해당 도메인 질문이 있으면 재사용, 없으면 소규모 수작업 질문셋 제작)
- 서브4(Phase 4.1~4.2)의 QA 평가 하네스를 재사용해 zero-shot vs adapted 모델의 다운스트림 QA 정확도 비교
- **산출물**: 도메인 적응 전/후 QA 정확도 비교표 — 논문 기여 C2의 핵심 근거 (triple 지표만으로는 "도메인 적응이 실제 검색 품질에 도움되는가"에 답할 수 없었음)

## 전체 Phase 의존관계 요약

```
Phase 0:  0.0(로컬교사서빙) → 0.5(baseline 5종 재현, 로컬LLM 연동) ─────────────┐
                │                                                              │
서브1:          └→ 1.0→1.1→1.2→1.3→1.4→1.5→1.6                                │
                                  │                                            │
서브2:                            └→ 2.0→2.1→2.2→2.3→2.4→2.5→2.6              │
                                                            │                  │
서브3:                                                      └→ 3.0→3.1→3.2→3.3→3.35→3.4→3.5→3.6→3.7
                                                            │                        │        │
서브4:                                                      │                        │        └→ 4.1→4.2→4.3→4.4→4.5
                                                            │                        │                 ↑
서브5:                                                      └→ 5.0→5.1→5.2→5.3→5.4 ─┴─────────────────┘
                                                                                      (5.4는 서브4 하네스 재사용)
```

- **Phase 0(0.0, 0.5)은 1주차에 서브1과 완전 병렬로 착수** — baseline 재현 리스크를 최대한 일찍 드러내기 위함
- 서브1(1.0~1.6)과 서브2 초반(2.0~2.1) 준비는 코퍼스만 있으면 병렬 착수 가능
- 서브3은 서브2.6(최종 체크포인트) 확정 후 시작
- 서브3.35(검색 인터페이스)는 서브3.3(그래프 DB) 스키마 확정에 영향을 주므로 동시 설계 권장
- 서브3.7(커버리지 recall)은 서브1의 고정 평가 서브셋 산출물이 필요
- 서브5는 서브2.6 완료 즉시 서브3과 병렬 진행 가능 (인력 여유 시), 단 Phase 5.4는 서브4 하네스가 준비된 후 실행
- 서브4의 QA 실행(4.1~4.5)은 서브3.35(검색 인터페이스), 서브3.7(그래프 완성) 이후 시작 — baseline 자체 재현(Phase 0.5)은 이미 끝나 있어야 함
