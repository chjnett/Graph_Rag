# LLM-Free GraphRAG 증류 프로젝트 — 서브프로젝트 분할 설계 (접근 가능 데이터셋 기반)

전체 프로젝트를 **독립적으로 진행/검증 가능한 5개 서브프로젝트**로 쪼갰습니다. 각 서브프로젝트는 그 자체로 결과물(중간 산출물 or 미니 실험)을 내고, 순서대로 이어붙이면 최종 논문이 완성되는 구조입니다. 모든 데이터셋은 로그인/게이트 없이 접근 가능한 공개 데이터셋 위주로 선정했습니다.

> **리소스 전제**: 단일 RTX 3090(24GB, 시간 무제한) + API 예산 사실상 $0. "교사 LLM"은 아래 전 항목에서 로컬 vLLM 서빙 강모델(예: Qwen2.5-32B-Instruct-AWQ)을 의미하며, baseline 인덱싱도 동일 로컬 모델로 재현합니다. 상세는 `graphrag_llmfree_design.md` 상단 및 §5.1/§6.2 참고.

## 데이터셋 인벤토리 (먼저 확정)

| 데이터셋 | 용도 | 접근 경로 | 비고 |
|---|---|---|---|
| **UltraDomain** | 원본 코퍼스 (그래프 구축 대상) | HuggingFace `TommyChien/UltraDomain` | 428개 대학교재, 18개 도메인, GraphRAG/LightRAG/MemoRAG 등에서 표준으로 쓰는 벤치마크라 baseline과 공정 비교 가능. Fin(금융)/Leg(법률) 서브셋 포함되어 있어 도메인 적응 실험에도 바로 활용 |
| **GraphRAG-Bench** | 통합 평가 (그래프 구축~QA 전체 파이프라인) | HuggingFace `GraphRAG-Bench/GraphRAG-Bench` | 난이도별 질문(사실검색/복합추론/요약/생성) 태깅 있어 세분화된 평가 가능 |
| **HotpotQA** | 멀티홉 QA 다운스트림 평가 | HuggingFace `hotpot_qa` | 정답/근거 문단(supporting facts) 포함 → 인용 정확도 평가에 유용 |
| **MultiHop-RAG** | 멀티홉 QA 평가 (뉴스 도메인) | HuggingFace `yixuantt/MultiHopRAG` | UltraDomain과 다른 도메인이라 일반화 검증용 |
| **DocRED** | 학생 모델 relation extraction 학습/평가 (정답 라벨 有) | GitHub `thunlp/DocRED`, HuggingFace `docred` | 원격지도(distant supervision) + 사람 검수 라벨 존재 → 증류 없이도 "정답 있는" 절대 성능 상한선 측정 가능. **단, DocRED는 96종 닫힌 Wikidata 관계 스키마라 우리 교사 모델의 자유 관계어(open-vocabulary)와 직접 비교 불가 — 반드시 관계어 정렬(mapping) 단계를 거쳐야 유효한 비교가 됨 (서브1 참고)** |
| **SciERC** | 학생 모델 보조 학습 데이터 (일반화) | HuggingFace `sciERC` / 공식 사이트 | 과학 논문 도메인 엔티티/관계 라벨 |
| **WebNLG** | 학생 모델 보조 학습 데이터 | HuggingFace `web_nlg` | (엔티티, 관계, 텍스트) 트리플이 이미 정제되어 있어 필터링 로직 검증용으로도 사용 가능 |
| **FewRel** | 저자원 관계 유형 일반화 테스트 | HuggingFace `few_rel` | 학생 모델이 학습 데이터에 없던 관계 유형에 얼마나 강건한지 확인 |

## 서브프로젝트 1 — 합성 데이터 생성 & 검증 파이프라인

**목표**: 교사 LLM으로 (문서 청크 → triple) 합성 데이터를 만들고, 품질을 정량적으로 검증

**데이터셋**: UltraDomain 원문 청크 (교사 LLM 입력) + DocRED (교사 LLM이 만든 triple의 정확도를 비교할 "정답이 있는" 대조군)

**작업**:
1. UltraDomain 청크 샘플 500~1,000개에 대해 로컬 교사 모델(vLLM 서빙)로 triple 생성 (API 비용 없음, GPU 시간만 소요)
2. 동일한 방식으로 DocRED 문서에도 교사 모델을 돌려서, **교사가 만든 자유 관계어를 DocRED 96종 스키마로 정렬(mapping)한 뒤** DocRED 정답 라벨과 비교 → 교사 모델 자체의 추출 정확도(상한선) 정량화. 정렬 없이 바로 비교하면 스키마 불일치로 점수가 인위적으로 낮게 나와 무의미함
3. 필터링 규칙 적용 전/후 노이즈 비율 비교
4. **(신설, 리뷰 반영)** 고정 평가 서브셋을 도메인별 균등 샘플 **150~200청크**(도메인당 약 8~11개, 기존 50~100청크에서 확대 — 표본 크기 문제 보완)로 지정하고, 그중 30~50 triple을 무작위 층화 추출해 **저자 본인이 원문 `source_span` 대조로 직접 gold 라벨링** — 이후 서브프로젝트 3에서 "교사-학생 일치율"과 "실제 정확도" 두 지표를 분리 산출하기 위한 선행 작업 (순환논증 방지, 단일 annotator 한계는 산출물 문서에 명시)

**산출물**: 합성 학습 데이터셋(JSON) + 데이터 품질 리포트 (정밀도/재현율 vs DocRED, 관계어 정렬 매핑 테이블 포함) + 고정 평가 서브셋(150~200청크) + 인간 검수 골드셋(30~50 triple, 라벨 포함)

## 서브프로젝트 2 — 경량 추출 모델 SFT (증류)

**목표**: 서브프로젝트 1의 합성 데이터로 경량 모델을 학습시켜, LLM 없이 triple을 추출하는 모델 확보

**데이터셋**: 서브프로젝트 1 산출물(주 학습 데이터) + SciERC, WebNLG, FewRel (일반화를 위한 보조 학습 데이터, 비율 실험 필요) + DocRED test split (최종 평가용, 라벨 공개되어 있어 외부 재현 가능)

**작업**:
1. 소형 모델(DeBERTa-v3 또는 1B급 seq2seq) QLoRA/LoRA SFT
2. DocRED test set으로 F1 평가 → "LLM 없이 학습된 모델이 공식 라벨 데이터에서 어느 정도 성능인지" 논문에 넣을 수 있는 **외부 검증 가능한 수치**
3. FewRel로 미학습 관계 유형에 대한 일반화 테스트

**산출물**: 학생 모델 체크포인트 + DocRED/FewRel 벤치마크 성능표

## 서브프로젝트 3 — 그래프 구축 & 커뮤니티 요약

**목표**: 학생 모델로 실제 대규모 코퍼스에서 그래프를 구축하고, LLM-free 커뮤니티 요약까지 통합

**데이터셋**: UltraDomain 전체 (여러 baseline 논문이 이미 이 데이터로 그래프를 구축해놓은 사례가 많아 구조적 통계 비교 가능 — 예: 앞서 확인한 논문들에서 UltraDomain 그래프의 노드/엣지 수치가 보고되어 있음)

**작업**:
1. 학생 모델 추론으로 전체 코퍼스에 대해 triple 추출 → 그래프 구축
2. 엔티티 중복 제거 (임베딩 유사도)
3. Leiden/Louvain 커뮤니티 탐지 + 추출 요약(TextRank 등)
4. 우리 그래프의 노드 수/엣지 수/평균 degree를 기존 논문에 보고된 UltraDomain 그래프 통계와 비교 → "LLM 기반 그래프와 구조적으로 얼마나 유사한가" 정량화
5. **고정 평가 서브셋**(도메인별 균등 샘플 150~200청크, 서브1에서 이미 교사 모델로 참조 triple을 만들어둔 범위와 겹치게 선정)에 대해 학생 모델 그래프 vs 교사 모델 참조 그래프의 엔티티/관계 recall(**교사-학생 일치율**)을 계산 — 이게 있어야 설계서 §6.3의 "품질(교사-학생 일치)" 지표가 실제로 산출됨(구조 통계 비교만으로는 recall을 대신할 수 없음)
6. **(신설, 리뷰 반영)** 서브1에서 만든 **인간 검수 골드셋(30~50 triple)** 대비 학생 모델의 실제 precision/recall 계산 — 5번(교사-학생 일치율)과 별도 지표로 분리해 두 수치의 괴리를 확인. 교사-학생 일치율은 높은데 골드셋 정확도가 낮다면 교사 모델의 체계적 편향이 학생에게 전이됐다는 증거이며, 이는 순환논증 없이 실제 품질을 주장하기 위한 필수 검증 단계
7. 쿼리 → 서브그래프 탐색(1-hop/2-hop) → 컨텍스트 구성하는 검색(retrieval) 인터페이스 구축 — 서브4 QA 평가에 필수 선행 요소이며 그래프 DB 스키마(작업 3의 NetworkX/Neo4j 설계) 확정 전에 요구사항으로 반영해야 함

**산출물**: 그래프 DB (NetworkX/Neo4j) + 구조 통계 비교표 + 교사-학생 일치율 리포트 + 골드셋 실제 정확도 리포트 + 검색 인터페이스 코드

## 서브프로젝트 4 — 다운스트림 QA 평가 (전체 시스템 비교)

**목표**: 우리 파이프라인 vs LLM 기반 GraphRAG/LightRAG를 실제 QA 정확도로 비교

**데이터셋**: GraphRAG-Bench (난이도별 질문 有), HotpotQA, MultiHop-RAG (도메인 일반화 검증)

**작업**:
0. **(일정 변경) Baseline 5종 재현은 서브1과 병렬로 1주차부터 조기 착수** — 외부 코드 재현이 전체 프로젝트에서 가장 시간 예측이 어려운 작업이라 서브3(그래프 구축) 완료를 기다리지 않고 최대한 일찍 리스크를 확인. Microsoft GraphRAG/LightRAG의 인덱싱 LLM 호출은 **로컬 vLLM 서빙 모델(서브1과 동일 모델)을 OpenAI 호환 엔드포인트로 연결**해 대체 — API 비용 없이 재현
1. GraphRAG-Bench의 4단계 난이도(사실검색/복합추론/맥락요약/창의생성)별로 우리 방법과 baseline 성능 분리 측정
2. HotpotQA supporting facts로 인용/근거 일치율 측정
3. MultiHop-RAG로 뉴스 도메인 일반화(=UltraDomain 밖 도메인) 확인
4. **(신설, 리뷰 반영) baseline 원 논문(Microsoft GraphRAG/LightRAG 등)이 GPT-4 계열로 공개 보고한 정확도 수치를 조사·정리해 참고 앵커로 최종 Table에 병기** — 우리 baseline 재현은 로컬 LLM 기반이라 원 논문과 직접 비교가 아니므로, 이 앵커 없이는 "LLM 기반과 동등"이라는 주장이 "같은 약한 교사를 쓴 두 방법 간 비교"로 축소될 위험이 있음

**산출물**: 논문 Table (baseline 대비 정확도/GPU-시간 비교 + 원 논문 보고 수치 앵커 병기), 난이도별 성능 분해 그래프

## 서브프로젝트 5 — 도메인 적응 실험

**목표**: 범용 학생 모델 vs 도메인 특화 파인튜닝 모델 비교 (JD의 "도메인 특화 파인튜닝" 요구사항 직접 대응)

**데이터셋**: UltraDomain 내 **Fin(금융)**, **Leg(법률)** 서브셋 (이미 공개되어 있고 도메인이 명확히 분리되어 있어 별도 도메인 데이터 수집 불필요)

**작업**:
1. 범용 데이터로 학습된 서브프로젝트 2 모델을 그대로 Fin/Leg에 적용 (zero-shot 성능)
2. Fin/Leg 서브셋 일부로 추가 파인튜닝 후 재평가
3. 성능 격차(zero-shot vs adapted) 정량화 — triple 추출 레벨
4. **Fin/Leg 도메인 특화 질문셋으로 서브4의 QA 평가 하네스를 재사용**해 zero-shot vs adapted 모델의 다운스트림 QA 정확도까지 비교 — 논문 기여 C2("도메인 적응 전략")의 실질적 의미는 결국 QA 품질 개선이므로, triple 지표만으로 끝내면 안 됨

**산출물**: 도메인 적응 전/후 성능 비교표 (triple 레벨 + QA 레벨 둘 다) — 논문의 핵심 ablation 중 하나

## 서브프로젝트 간 의존관계

```
[서브1: 합성데이터] → [서브2: 학생모델 학습] → [서브3: 그래프구축] → [서브4: QA평가]
                                    └──────────→ [서브5: 도메인적응] ──┘
[서브4 Baseline 재현(로컬 LLM)] ──────────────────────────────────────┘  (1주차부터 서브1과 병렬 착수)
```

서브1~2는 독립적으로 먼저 진행 가능(코퍼스만 있으면 됨), 서브3부터는 서브2의 체크포인트가 필요합니다. 서브5는 서브2 완료 후 병렬로 진행 가능해서, 인력이 여유 있다면 서브3/4와 동시에 진행하는 것도 가능합니다. **단, 서브4의 baseline 재현(로컬 LLM 연동 포함) 자체는 서브3 완료를 기다릴 필요가 없어 최대한 일찍(1주차) 시작**합니다 — QA 평가 실행만 서브3.6 이후로 미루면 됩니다.

## 왜 이 데이터셋 조합이 유리한가

1. **접근성**: 전부 HuggingFace/GitHub에서 로그인 없이 바로 받을 수 있음 (TACRED처럼 LDC 라이선스가 필요한 데이터는 의도적으로 제외)
2. **비교 가능성**: UltraDomain, GraphRAG-Bench는 이미 GraphRAG 계열 논문들의 사실상 표준 벤치마크라, 우리 결과를 기존 논문 수치와 나란히 놓고 비교하기 쉬움
3. **외부 검증 가능성**: DocRED처럼 정답 라벨이 있는 데이터를 끼워 넣어서, "우리 모델이 실제로 맞는 관계를 뽑는가"를 LLM-as-judge 없이도 객관적으로 증명 가능
4. **도메인 다양성 확보**: UltraDomain 자체에 이미 Fin/Leg 서브셋이 있어서 별도 도메인 데이터 수집 없이 도메인 적응 실험(서브5)이 바로 가능
