# 서브프로젝트 3 — 그래프 구축 & 커뮤니티 요약

> 리소스 전제는 `graphrag_00_overview.md` 참고.

## 목표
학생 모델로 실제 대규모 코퍼스에서 그래프를 구축하고, LLM-free 커뮤니티 요약까지 통합한다.

## 데이터셋
UltraDomain 전체 (여러 baseline 논문이 이미 이 데이터로 그래프를 구축해놓은 사례가 많아 구조적 통계 비교 가능)

## 설계 개요
- 경량 모델 추론 → triple 수집 → 엔티티 중복 제거(entity resolution, 임베딩 유사도 기반)
- NetworkX/Neo4j 그래프 저장, (선택) 커뮤니티 탐지: Leiden/Louvain 알고리즘(LLM 없이)
- 커뮤니티 요약: 기존 연구 대부분 LLM 의존적 → 추출적 요약(extractive summarization, TextRank 등) 또는 소형 모델로 대체 (논문의 니치 포인트)

> **개정 (methodology 리뷰 반영, 순환논증 방지)**: 기존 "그래프 커버리지 recall" 단일 지표는 학생 모델의 학습 신호(교사 출력)와 평가 정답(교사 출력)이 동일해 순환논증 위험이 있었다. 아래 Phase 3.7에서 이를 **"교사-학생 일치율"**과 **"골드셋 실제 정확도"** 두 지표로 분리했다.

## Phase 3.0 — 추론 파이프라인 구축
- **3.0-a** (M) 배치 추론 스크립트 (긴 문서 청킹 전략 포함), GPU 배치 크기 튜닝
- **3.0-b** (S) 처리량 벤치마크 (문서/초, GPU 메모리 사용량) 기록 — 비용 비교 근거자료로 재사용
- Done when: 10개 문서로 end-to-end 추론 smoke test 통과
- **산출물**: 추론 파이프라인 코드

## Phase 3.1 — 전체 코퍼스 Triple 추출
- **3.1-a** (L) UltraDomain 전체(428개 교재) 배치 실행, 중간 체크포인트/재개 지원
- **3.1-b** (S) 원시 triple에 문서/도메인 메타데이터 태깅, Phase 1.2-f 고정 평가 서브셋 문서는 별도 플래그 표시
- Done when: 전체 코퍼스 triple 원시 데이터셋 완성
- **산출물**: 원시 triple 대규모 데이터셋

## Phase 3.2 — 엔티티 정규화 & 중복 제거
- **3.2-a** (S) 문장 임베딩 모델 선정 및 엔티티 문자열 임베딩 생성
- **3.2-b** (M, 1단계 방어선 — 과묶임 방지) 유사도 기반 클러스터링/병합 로직 구현: **entity_type이 다르면 병합 후보에서 먼저 제외**한 뒤 그 안에서만 임베딩 유사도 검사, 임계값 스윕으로 최적점 탐색 (예: 애플-기업과 사과-과일은 type이 달라 애초에 병합 후보군에 안 들어감)
- **3.2-b2** (S, 신규, 2단계 방어선 — 과소 병합 방지) 3.2-b를 통과한 동일 type 후보 내에서, 순수 임베딩 유사도만으로는 놓치는 표기 변형(예: "Steve Jobs" vs "Jobs", 이니셜/부분 문자열)을 잡기 위한 규칙 기반 별칭 매칭 보조 — 성(last name) 단독 언급이 앞서 등장한 전체 이름의 부분 문자열이면서 같은 문서/근접 문맥에 나타나는 경우 후보로 승격, 최종 병합은 여전히 임베딩 유사도 임계값과 함께 판정(별칭 매칭 단독으로 자동 병합하지 않음 — 오탐 방지)
- **3.2-c** (S) 병합 오류 샘플 검수 (over-merge/under-merge 사례 육안 확인 20건, 3.2-b2 도입 전/후 under-merge 비율 비교)
- Done when: 정규화된 엔티티 매핑 테이블 + 병합 정밀도 수기 검증 결과 + 3.2-b2 적용 전후 under-merge 개선 여부 기록
- **산출물**: 정규화된 엔티티 매핑 테이블 + 별칭 매칭 규칙 문서

## Phase 3.3 — 그래프 DB 구축
- **3.3-a** (S) Phase 3.35(검색 인터페이스)에 필요한 쿼리 패턴(1-hop/2-hop 탐색, 커뮤니티 조회)을 먼저 목록화한 뒤 NetworkX 스키마(노드/엣지 속성) 확정
- **3.3-b** (M) Triple → 그래프 적재 스크립트, 대규모 시 Neo4j 마이그레이션 스크립트
- Done when: 그래프 인스턴스 로드 + 기본 통계(노드/엣지 수) 출력 확인
- **산출물**: 그래프 DB 인스턴스 + 로딩 스크립트

## Phase 3.35 — 검색(retrieval) 인터페이스 구축
- **3.35-a** (M) 쿼리 → 서브그래프 탐색(1-hop/2-hop) 로직 구현
- **3.35-b** (M) 서브그래프 랭킹 기준 구현 (엔티티 매칭 점수, degree 기반 중요도 등)
- **3.35-c** (S) 컨텍스트 구성(서브그래프 → 텍스트 프롬프트 변환) 로직
- **3.35-d** (S) 최종 응답 생성은 로컬 LLM 재활용(또는 별도 응답 생성기) — 인덱싱만 LLM-free임을 코드 주석/문서에 명시. 이 부분은 GraphRAG의 표준 구조와 동일
- Done when: 쿼리 1건에 대해 end-to-end(쿼리→서브그래프→응답) 테스트 통과
- **산출물**: 검색 인터페이스 코드, 쿼리 1건 end-to-end 테스트 통과
- **왜 필요한가**: `graphrag_04_evaluation.md`의 QA 평가가 성립하려면 반드시 필요한 선행 요소

## Phase 3.4 — 커뮤니티 탐지
- **3.4-a** (S) Leiden/Louvain 파라미터(resolution) 스윕
- **3.4-b** (S) 커뮤니티 크기 분포 시각화, 지나치게 크거나 작은 커뮤니티 확인
- Done when: 최종 resolution 값 확정 + 라벨링된 그래프 저장
- **산출물**: 커뮤니티 라벨링된 그래프

## Phase 3.5 — 커뮤니티 요약 생성
- **3.5-a** (M) 커뮤니티 소속 엔티티/관계가 언급된 원문 문장 풀을 구성한 뒤 TextRank(또는 대안) 추출 요약 구현 (커뮤니티=엔티티/관계 집합이므로 문장 풀로 변환하는 단계가 선행되어야 함)
- **3.5-b** (S) 커뮤니티별 요약 품질 수기 샘플 검토(10~15개) — LLM 요약과의 정성적 격차를 미리 파악해 Limitations에 반영할 근거 확보
- Done when: 커뮤니티별 요약 텍스트 저장 + 정성 검토 메모
- **산출물**: 커뮤니티별 요약 텍스트 + 정성 검토 메모

## Phase 3.6 — 구조 통계 비교
- **3.6-a** (S) 노드 수/엣지 수/평균 degree/커뮤니티 수 산출
- **3.6-b** (S) 기존 baseline 논문에 보고된 UltraDomain 그래프 통계와 나란히 비교표 작성
- **3.6-c** (S, 삼각비교) baseline 원 논문(Microsoft GraphRAG/LightRAG 등)이 GPT-4 계열로 보고한 QA/커버리지 수치(가능한 범위)를 별도 참고 열로 정리 — 우리 baseline 재현(로컬 LLM, `graphrag_04_evaluation.md` Phase 0.5)과 원 논문 수치 사이 괴리 판단용 삼각비교 자료, Phase 4.5에서 재사용
- Done when: 비교표 + 원 논문 앵커 열 + 구조적 유사성에 대한 짧은 해석 문단 (※ 이는 recall이 아니라 구조 비교임에 유의)
- **산출물**: 구조 통계 비교표 + 원 논문 수치 앵커 열

## Phase 3.7 — 교사-학생 일치율 & 골드셋 실제 정확도 계산
- **3.7-a** (S) `graphrag_01_data_pipeline.md` Phase 1.2-f 고정 평가 서브셋(150~200청크, 교사 모델 참조 triple)과 Phase 3.1의 학생 모델 triple 중 동일 서브셋을 매칭
- **3.7-b** (M) Phase 1.4와 동일한 매칭 규칙으로 엔티티 recall, 관계 recall 계산 → **교사-학생 일치율**로 명명 (교사 출력을 정답으로 쓰는 지표이므로 "실제 정확도"가 아님을 리포트에 명시 — 교사가 틀려도 학생이 복제하면 일치율은 높게 나옴)
- **3.7-c** (S) 도메인별 일치율 분포 확인, 단 도메인당 표본 10개 내외이므로 **탐색적(exploratory) 해석**으로 한정한다고 리포트에 명기
- **3.7-d** (M) Phase 1.6-d 인간 검수 골드셋(30~50 triple)과 Phase 3.1 학생 모델 triple을 동일 매칭 규칙(1.4 기준)으로 비교 → **골드셋 실제 정확도(precision/recall)** 산출
- **3.7-e** (S) 3.7-b(교사-학생 일치율)와 3.7-d(골드셋 실제 정확도)를 나란히 비교 — 괴리가 크면 교사 모델 편향이 학생에게 전이된 것으로 해석, Limitations/Discussion에 반영할 근거로 기록
- Done when: 교사-학생 일치율 리포트 + 골드셋 실제 정확도 리포트 + 두 지표 비교 분석 완성 — 순환논증 없이 실제 품질 주장 가능
- **산출물**: 교사-학생 일치율 리포트 + 골드셋 실제 정확도 리포트 + 두 지표 비교 분석
- **왜 필요한가**: Phase 3.6의 구조 통계 비교만으로는 "같은 관계를 뽑았는가"를 답할 수 없고, 교사-학생 일치율만으로는 "그게 실제로 맞는가"를 증명할 수 없음(순환논증). 두 지표를 함께 산출해야 "실제 품질 유지" 주장이 성립함

## 코드 매핑

```
src/graph_construction/
├── extractor.py           # 학생 모델 추론 wrapper (Phase 3.0~3.1)
├── entity_resolution.py   # 임베딩 기반 엔티티 중복 제거 + type 사전필터(과묶임 방지) + 별칭 매칭 보조(과소병합 방지) (Phase 3.2, 3.2-b2)
├── graph_builder.py       # NetworkX/Neo4j 그래프 생성 (Phase 3.3)
└── community_summary.py   # LLM-free 요약, TextRank 등 (Phase 3.4~3.5)
src/retrieval/
└── graph_retriever.py     # 쿼리 시 그래프 탐색/서브그래프 추출, 1-hop/2-hop, 랭킹 (Phase 3.35)
```

### 기술 스택
| 영역 | 도구 |
|---|---|
| 그래프 저장/탐색 | `networkx` (프로토타입), `Neo4j` (스케일업, 로컬 docker) |
| 커뮤니티 탐지 | `python-igraph`(Leiden), `networkx`(Louvain) |
| 임베딩(엔티티 중복 제거용) | `sentence-transformers` (로컬 실행) |

## 이 파트의 한계
- TextRank 기반 커뮤니티 요약은 LLM 서술형 요약 대비 품질이 열위일 가능성이 높음 (커뮤니티는 "문장 집합"이 아니라 "엔티티/관계 집합"이라 TextRank 적용 방식 자체가 근사적임)
- **도메인별 세부 분석(18개 도메인)은 도메인당 표본이 10개 내외로 작아 탐색적(exploratory) 해석에 한정** — 도메인 간 차이에 대한 확증적 결론은 유보

## 산출물 총괄
그래프 DB (NetworkX/Neo4j) + 구조 통계 비교표(원 논문 앵커 포함) + 교사-학생 일치율 리포트 + 골드셋 실제 정확도 리포트 + 검색 인터페이스 코드
